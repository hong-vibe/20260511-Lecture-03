# GitHub 저장소 검색 기능 구현 프롬프트 기록

이 문서는 AI와 함께 GitHub REST API를 활용한 저장소 검색 기능을 구현하며 진행했던 대화와 요청 사항(프롬프트)들을 시간순으로 정리한 기록입니다.

## 1단계: API URL 설계 및 기본 Fetch 예시 요청
**요청 내용:**
GitHub REST API에서 저장소를 검색하는 요청 URL을 만들고 싶습니다.
조건:
1. 검색어는 사용자가 입력한 keyword를 사용합니다.
2. 저장소 검색 API를 사용합니다.
3. stars 기준으로 정렬합니다.
4. 내림차순으로 정렬합니다.
5. 결과는 6개만 가져옵니다.
6. fetch에서 사용할 수 있는 URL 문자열 예시를 보여주세요.

**진행 결과:**
조건에 맞는 GitHub Search API(`https://api.github.com/search/repositories`) URL의 구조와 간단한 `fetch` 사용 예제를 제공받았습니다.

## 13단계: async/await 기반 fetchRepos 함수 작성 및 적용
**요청 내용:**
GitHub Repository Search API를 호출하는 `fetchRepos` 함수를 작성해 주세요. (조건: keyword 매개변수, async/await 사용, encodeURIComponent 처리, stars 내림차순/6개 제한, response.ok 확인 및 에러 처리, JSON 반환, 초보자용 주석 추가)

**진행 결과:**
데이터 요청 로직을 별도의 비동기 함수인 `fetchRepos`로 분리했습니다. `fetch`와 `async/await` 문법을 사용하여 가독성을 높였으며, `response.ok` 체크와 `throw new Error`를 통해 예외 상황을 견고하게 처리하도록 구현했습니다. 기존의 `searchRepositories` 함수도 이 `fetchRepos`를 호출하여 데이터를 비동기적으로 받아오도록 리팩토링했습니다.

## 16단계: handleSearch 로직 통합 및 함수명 변경 (renderRepos)
**요청 내용:**
`handleSearch` 함수에서 `fetchRepos`로 가져온 데이터의 `items` 배열을 `renderRepos` 함수에 전달하도록 코드를 수정해 주세요. (조건: `const data = await fetchRepos(keyword)` 사용, `renderRepos(data.items)` 호출, 기존 입력값 검증 유지 등)

**진행 결과:**
중간 매개 역할을 하던 `searchRepositories` 함수를 제거하고, `handleSearch` 함수가 직접 데이터 요청(`fetchRepos`)과 화면 렌더링(`renderRepos`)을 제어하도록 구조를 통합했습니다. 요청에 따라 `renderCards`의 이름을 `renderRepos`로 변경하였으며, `async/await`를 활용해 비동기 흐름을 한곳에서 관리하도록 리팩토링했습니다.

## 17단계: 검색 실행 흐름 고도화 (로딩/결과없음/에러 처리 및 버튼 제어)
**요청 내용:**
검색 실행 흐름에 `try/catch/finally`를 적용하고, 로딩 상태("검색 중입니다..."), 결과 없음, 에러 처리 메시지를 보완해 주세요. 또한 검색 중에는 버튼을 비활성화하고 이전 결과를 비우도록 기능을 추가해 주세요.

**진행 결과:**
`handleSearch` 함수에 `finally` 블록을 추가하여 검색 완료 후 버튼 활성화 상태를 보장하도록 했습니다. 검색 중 버튼 비활성화(`disabled = true`)와 결과 목록 초기화 로직을 추가하여 중복 요청을 방지하고 UI 일관성을 높였습니다. 상황별(로딩, 결과 없음, 에러)로 더욱 명확한 한국어 안내 메시지를 출력하도록 수정하고 초보자용 주석을 보강했습니다.
